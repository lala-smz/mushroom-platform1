# 时令菌菇盲盒系统 API 文档

## 1. 系统概述

时令菌菇盲盒系统是一个基于前后端分离架构的应用，允许用户抽取包含不同种类菌菇的盲盒，并提供完整的管理后台功能。

## 2. 数据模型变更

### 2.1 Mushroom 模型

**新增字段：**
- `type`：菌菇类型，区分常见菌菇和自有商品菌菇
  - 类型：VARCHAR(20)
  - 默认值：'common'
  - 可选值：'common'（常见菌菇）, 'product'（自有商品菌菇）
  - 注释：菌菇类型（common: 常见菌菇, product: 自有商品菌菇）

### 2.2 UserBoxOrder 模型

**新增字段：**
- `totalPrice`：订单总价
  - 类型：DECIMAL(10, 2)
  - 默认值：0
  - 注释：销售统计使用

## 3. API 接口

### 3.1 盲盒管理接口

#### 3.1.1 获取盲盒列表
- **URL**：`/api/mushroom-boxes`
- **方法**：GET
- **权限**：无（公开接口）
- **参数**：无
- **返回**：盲盒列表，包含所有盲盒配置

#### 3.1.2 获取盲盒详情
- **URL**：`/api/mushroom-boxes/:id`
- **方法**：GET
- **权限**：无（公开接口）
- **参数**：
  - `id`：盲盒 ID
- **返回**：盲盒详细信息，包含内容配置

#### 3.1.3 添加新盲盒配置
- **URL**：`/api/mushroom-boxes`
- **方法**：POST
- **权限**：管理员
- **参数**：
  - `name`：盲盒名称
  - `description`：盲盒描述
  - `price`：盲盒价格
  - `stock`：盲盒库存
  - `image`：盲盒图片
  - `content`：盲盒内容配置（包含菌菇 ID 和概率）
- **返回**：创建成功的盲盒信息

#### 3.1.4 更新盲盒配置
- **URL**：`/api/mushroom-boxes/:id`
- **方法**：PUT
- **权限**：管理员
- **参数**：
  - `id`：盲盒 ID
  - `name`：盲盒名称（可选）
  - `description`：盲盒描述（可选）
  - `price`：盲盒价格（可选）
  - `stock`：盲盒库存（可选）
  - `image`：盲盒图片（可选）
  - `content`：盲盒内容配置（可选）
- **返回**：更新后的盲盒信息

#### 3.1.5 删除盲盒配置
- **URL**：`/api/mushroom-boxes/:id`
- **方法**：DELETE
- **权限**：管理员
- **参数**：
  - `id`：盲盒 ID
- **返回**：删除成功信息

### 3.2 盲盒抽取接口

#### 3.2.1 抽取盲盒
- **URL**：`/api/mushroom-boxes/:id/draw`
- **方法**：POST
- **权限**：登录用户
- **参数**：
  - `id`：盲盒 ID
- **返回**：抽取结果，包含获得的菌菇信息

### 3.3 数据统计接口

#### 3.3.1 获取盲盒统计数据
- **URL**：`/api/mushroom-boxes/statistics`
- **方法**：GET
- **权限**：管理员
- **参数**：无
- **返回**：
  - `salesTotal`：销售总额
  - `salesCount`：销售数量
  - `stockTotal`：库存总量
  - `boxes`：各盲盒的详细统计数据

### 3.4 菌菇管理接口

#### 3.4.1 获取菌菇列表（带类型筛选）
- **URL**：`/api/mushrooms`
- **方法**：GET
- **权限**：无（公开接口）
- **参数**：
  - `type`：菌菇类型（可选，'common' 或 'product'）
- **返回**：菌菇列表

## 4. 数据同步机制

### 4.1 前端数据同步

前端使用 Pinia 状态管理库实现数据同步：

1. **定期同步**：每 30 秒自动同步一次盲盒数据
2. **手动同步**：提供手动同步按钮，允许用户随时触发同步
3. **事件驱动同步**：在盲盒抽取、管理操作后自动触发同步

### 4.2 后端数据一致性

后端通过以下机制确保数据一致性：

1. **事务处理**：所有库存变更操作使用数据库事务
2. **并发控制**：使用数据库锁机制防止并发操作冲突
3. **数据验证**：严格的数据验证确保数据完整性

## 5. 系统安全

### 5.1 API 安全

- **JWT 认证**：所有需要权限的接口使用 JWT 令牌认证
- **权限控制**：基于角色的权限控制，区分普通用户和管理员
- **输入验证**：所有用户输入进行严格验证，防止 SQL 注入和 XSS 攻击

### 5.2 数据安全

- **敏感数据加密**：密码等敏感数据使用 bcrypt 加密存储
- **数据备份**：定期进行数据库备份
- **访问日志**：记录所有 API 访问日志，便于审计和故障排查

## 6. 性能优化

### 6.1 数据库优化

- **索引优化**：为频繁查询的字段添加索引
- **查询优化**：使用分页查询，避免一次性加载大量数据
- **缓存机制**：使用 Redis 缓存热点数据

### 6.2 API 优化

- **响应压缩**：启用 gzip 压缩，减少传输数据量
- **批量操作**：支持批量 API 操作，减少网络请求次数
- **异步处理**：使用异步非阻塞 I/O，提高并发处理能力

## 7. 错误处理

### 7.1 错误码定义

- `400`：请求参数错误
- `401`：未授权
- `403`：禁止访问
- `404`：资源不存在
- `500`：服务器内部错误

### 7.2 错误响应格式

```json
{
  "code": 400,
  "message": "错误信息"
}
```

## 8. 部署说明

### 8.1 环境要求

- Node.js >= 16.0.0
- MySQL >= 5.7.0
- Redis >= 6.0.0（可选，用于缓存）

### 8.2 数据库迁移

执行以下脚本更新数据库结构：

```bash
node migrate-mushroom-type.js
```

### 8.3 启动服务

```bash
# 安装依赖
npm install

# 启动服务
node app.js
```

服务默认运行在 `http://localhost:3002`
